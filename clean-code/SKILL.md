# 代码整洁之道（Clean Code）编写指南

基于 Robert C. Martin《Clean Code》笔记整理的中文版代码编写指南，作为编写、审查、重构代码时的对照标准。

## 适用场景

- 编写新代码时检查命名、函数大小、注释、错误处理是否符合整洁代码规范
- 代码审查（Code Review）时按章节清单逐项核对
- 重构遗留代码（坏味道识别与改进方向）
- 团队约定整洁代码的最低基线
- 单元测试质量评估（FIRST 原则）

## 核心信念

> 整洁的代码 = 易于阅读 + 经过精心维护的代码

- **童子军规则**：始终让营地比你发现时更干净
- **专业责任**：代码是程序员的责任，不能因为赶工期而牺牲整洁度
- **唯一加速方式**：保持代码尽可能整洁，是真正快速前进的唯一办法

## 一、命名（Naming）

命名应当**一针见血**：表达「为什么存在、做什么、如何使用」。如果一个名称需要注释解释，那它就没表达清楚意图。

### 命名清单

- [ ] **使用有意义的命名**：`int elapsedTimeInDays` 优于 `int d; // 经过的天数`
- [ ] **避免误导**：不要把非 List 容器命名为 `accountList`；避免形状相似的名称（`XYZControllerForEfficientHandlingOfStrings` vs `…StorageOfStrings`）
- [ ] **做有意义的区分**：用 `source/destination` 替代 `a1/a2`；避免噪声词 `Info`、`Data`、`variable`、`table`
- [ ] **使用好发音的命名**：`generationTimestamp` 优于 `genymdhms`
- [ ] **使用便于搜索的命名**：避免单字母名称和魔法数字
- [ ] **避免编码与前后缀**：避免匈牙利命名法、成员前缀；接口不要加 `I`，必要时给实现加 `Impl`
- [ ] **避免思维映射**：不要让读者把名称在脑中翻译成另一个名称
- [ ] **类名为名词**：`Customer`、`WikiPage`、`AddressParser`；避免 `Manager`、`Processor`、`Data`、`Info`
- [ ] **方法名为动词**：`postPayment`、`deletePage`、`save`；访问器使用 `get/set/is`
- [ ] **重载构造函数时使用静态工厂方法**：`Complex.FromRealNumber(23.0)` 优于 `new Complex(23.0)`
- [ ] **每个概念一个词**：`fetch/retrieve/get` 不要混用；不要一词多用（同一个 `add` 表达不同语义）
- [ ] **使用解决方案领域 / 问题领域名称**：CS 术语、设计模式名优先；没有合适程序员术语时用领域语言
- [ ] **添加有意义的上下文**：`addrState` 或封装为 `Address` 类
- [ ] **不要添加无意义的上下文**：避免给所有类加 `GSD` 前缀
- [ ] **不要玩梗**：`deleteItems` 优于 `holyHandGranade`

## 二、函数（Functions）

### 函数规则

- [ ] **小！** 第一规则：函数应该小。第二规则：应该更小
- [ ] **缩进不超过两层**：`if/else/while` 块内只有一行（通常是函数调用）
- [ ] **只做一件事**：无法合理划分为「声明、初始化、…」等小节
- [ ] **每个函数一个抽象层级**：遵循「逐步向下规则」（Stepdown Rule），让代码像「to 段落」自上而下阅读
- [ ] **Switch 语句**：藏在低层多态实现里，永不重复
- [ ] **使用描述性名称**：长而描述性的名称 > 短而晦涩的名称 > 长而描述性的注释

### 函数参数

理想参数数：**0 > 1 > 2 > 3**，不要超过 3 个。

- [ ] **避免标志参数**：布尔参数说明函数做了不止一件事，应拆成两个函数
- [ ] **避免输出参数**：通过返回值传出信息，或改变调用对象自身的状态
- [ ] **多参数 → 参数对象**：`makeCircle(Point center, double radius)` 优于 `makeCircle(double x, double y, double radius)`
- [ ] **动词/关键词形式**：`writeField(name)`、`assertExpectedEqualsActual(expected, actual)`

### 其他原则

- [ ] **命令与查询分离**（CQS）：函数要么改状态，要么读信息，不能兼有
- [ ] **优先抛异常而非返回错误码**
- [ ] **不要重复**（DRY）：重复是软件中所有邪恶的根源
- [ ] **结构化编程**：函数小时，多个 `return/break/continue` 比单一出口更具表现力；永远不用 `goto`

## 三、注释（Comments）

> 唯一真正好的注释，是足够清晰易懂的代码本身。

### 好的注释

- 法律注释（版权、作者声明）
- 信息性注释（基本信息说明）
- 意图解释（为什么这么做）
- 澄清（标准库或不能修改的代码）
- 警告后果（`// 除非你想浪费时间，否则不要运行`）
- TODO 注释（说明退化实现及未来计划）
- 放大（强调易被忽略的重要细节，如 `trim` 的必要性）
- 公共 API 的 Javadoc

### 糟糕的注释（要消除）

- [ ] 含糊不清的注释
- [ ] 冗余的注释（重复代码内容）
- [ ] 误导性注释（不精确）
- [ ] 强制要求的注释（每个函数都要 javadoc 的规定是愚蠢的）
- [ ] 日志式注释（用版本控制系统）
- [ ] 噪声注释（`/** 默认构造函数 */`）
- [ ] **能用函数或变量替代时不用注释**：`if (employee.isEligibleForFullBenefits())` 优于 `// 检查员工是否有资格…`
- [ ] 位置标记（`// Actions /////`）
- [ ] 闭合括号注释（`} // while`）—— 改成拆分小函数
- [ ] 署名作者信息（用版本控制系统）
- [ ] 注释掉的代码（删掉，需要时从版本控制恢复）
- [ ] HTML 注释
- [ ] 非局部信息 / 过多信息 / 不明显联系
- [ ] 短函数的函数头注释
- [ ] 非公共代码中的 Javadoc

## 四、格式化（Formatting）

### 垂直格式化

- [ ] **概念之间用空行分隔**
- [ ] **垂直密度**：紧密相关的代码行紧凑排列
- [ ] **垂直距离**：
  - 变量声明靠近使用位置
  - 实例变量统一放在类的顶部
  - 调用者在被调用者上方
  - 概念亲和的代码靠近
- [ ] **垂直顺序**：函数调用依赖关系从上到下流动

### 水平格式化

- [ ] **水平开放性与密度**：用空格分隔强相关与弱相关的元素（赋值号两侧）
- [ ] **避免水平对齐**：现代语言中，对齐强调错误的内容、分散注意力
- [ ] **缩进**：保持可见的层次结构

### 团队规则

> 团队规则优先于个人偏好。代码不应看起来像由意见不合的人编写。

## 五、对象与数据结构

### 数据抽象

- 类不仅是 getter/setter 暴露变量；它暴露**操作数据本质的抽象接口**

### 数据/对象的反对称性

| 范式 | 易于添加 | 难于添加 |
|---|---|---|
| 过程式（数据结构 + 函数） | 新函数 | 新数据结构 |
| 面向对象（多态对象） | 新类 | 新函数 |

> 一切皆对象是个神话。有时简单的数据结构 + 函数才是正确选择。

### 得墨忒耳原则（Law of Demeter）

类 `C` 的方法 `f` 只应调用以下对象的方法：

- `C` 自身
- `f` 创建的对象
- `f` 的参数
- `C` 的实例变量持有的对象

> 与朋友交流，而不是与陌生人交流。避免「火车失事」式链式调用。

### 数据传输对象（DTO）

- 公共变量、无函数的类
- 适用于数据库通信、套接字消息解析

## 六、错误处理（Error Handling）

错误处理重要，但不应该掩盖主逻辑。

- [ ] **使用异常而非错误码**
- [ ] **先编写 Try-Catch-Finally**：try 块像事务，从它开始定义异常路径下的一致状态
- [ ] **提供异常的上下文**：失败的操作、失败的类型、足以记录日志的信息
- [ ] **不要返回 null**：抛异常或返回特殊情况对象（Special Case）；包装第三方 API 的 null
- [ ] **不要传递 null**：除非 API 要求

---

## 七、边界（Boundaries）

干净地集成第三方代码。

- [ ] **封装第三方 API**：把 `Map`、强制类型转换隐藏在 `Sensors` 类里
- [ ] **学习测试（Learning Tests）**：编写小测试探索第三方 API 的行为
- [ ] **学习测试无本万利**：API 升级时回归验证行为差异
- [ ] **使用尚未存在的代码**：定义自己控制的接口隔离待开发模块
- [ ] **保持清晰的边界**：保护已有代码投资，让未来变更代价可控

---

## 八、单元测试（Unit Tests）

### TDD 三条法则

1. 不写失败测试就不写生产代码
2. 只写到足以失败的测试代码（编译不过也算失败）
3. 仅写让失败测试通过的最少生产代码

### 干净的测试

- 可读性 = 测试干净度
- **每个测试一个断言**（指导原则，非铁律）
- **每个测试一个概念**：为每个需要验证的概念写一个测试

### F.I.R.S.T 原则

- **F**ast（快速）
- **I**ndependent（独立，互不依赖）
- **R**epeatable（任何环境可重复）
- **S**elf-Validating（布尔输出：通过/失败）
- **T**imely（及时，先于生产代码）

## 九、类（Classes）

### 组织

- 类的顶部：公共静态常量 → 私有静态变量 → 私有实例变量 → 公共方法 → 私有方法（紧邻其调用者）
- 必要时为测试放宽到 protected

### 类应该小巧

- 第一规则：类应该小巧
- 第二规则：**类应该更小**
- 度量：职责数量（不是行数）

### 单一职责原则（SRP）

> 一个类只有一个引起它变化的原因

### 内聚性

- 实例变量少
- 每个方法操作多个实例变量
- 内聚性下降时，自然地拆分出新类

### 为变化而组织

- 隔离变化：依赖于抽象（接口/抽象类）而非具体细节
- 通过接口隔离让代码对变化更具弹性

## 十、系统（Systems）

### 构建与使用分离

- 启动逻辑（构建/连接对象）与运行时逻辑分离
- 把构建集中在 `main` 或其调用模块
- 抽象工厂模式是常见选择

### 依赖注入（DI）

- 控制反转（IoC）在依赖管理中的应用
- 对象不负责实例化其依赖
- 容器或 main 例程承担「权威」机制

---

## 十一、涌现的简单设计（Kent Beck）

按优先级遵循：

1. 运行所有测试
2. 不包含重复
3. 表达程序员的意图
4. 最小化类和方法的数量

---

## 十二、并发（Concurrency）

### 神话与误解

- ❌ 并发总是提高性能
- ❌ 编写并发程序时设计不会改变
- ❌ 使用容器时不必理解并发问题

### 平衡观点

- 并发会带来开销（性能与额外代码）
- 正确的并发即使对简单问题也复杂
- 并发错误通常不可重复，易被当作一次性问题忽视
- 并发常需要设计策略的根本性改变

## 使用方式

### 编写新代码时

1. 命名先行：变量/函数/类名是否一针见血？
2. 函数小巧：是否只做一件事？参数 ≤ 3？
3. 注释自省：能否用函数名/变量名替代注释？
4. 测试先行：TDD 三法则是否遵守？

### 代码审查时

按顺序对照：**命名 → 函数 → 注释 → 格式化 → 错误处理 → 测试**，逐项检查清单。

### 重构遗留代码时

按坏味道速查清单（C/E/F/G/N/T）扫描，确定优先改进点；先补特征测试再重构。
