# 24 种代码坏味道：识别 / 嗅探 / 治理

源自《重构》第 3 章（Beck & Fowler）。每条按统一格式：

- **症状**：直接可观察的征兆
- **嗅探问题**：背后隐藏的设计问题（更深层的"为什么坏"）
- **治理**：推荐手法（按优先级）
- **反例 / 何时不算坏味道**：避免矫枉过正

---

## 1. 神秘命名（Mysterious Name）

**症状**：函数/变量/类名看不出意图；要读实现才能用；缩写、单字母、`data`、`info`、`process`、`handle`、`manager` 等空名遍布。

**嗅探问题**：作者写时偷懒；或对自己在做什么不够清楚（往往伴随设计问题）。

**治理**：
1. 改变函数声明（包含改名）
2. 变量改名 / 字段改名
3. 想不出名字 → 这是更深的设计问题，先做提炼/合并

**何时不算**：循环内极短的临时变量（`i`、`j`、`x`、`y`）、众所周知的领域缩写（`xml`、`url`、`id`）。

---

## 2. 重复代码（Duplicated Code）

**症状**：

- 同一类两个函数中相同的表达式
- 同一继承体系不同子类中相同的代码块
- 不同模块间结构相似的代码（"看起来差不多"）

**嗅探问题**：每次修改都要找出所有副本。一旦漏改一处，就成 bug。

**治理**：
1. 同类内 → 提炼函数
2. 相似但不全同 → 先 移动语句 让相似部分对齐，再提炼
3. 同一超类的不同子类 → 函数上移
4. 不同类间 → 提炼超类 或 提炼成独立模块/工具函数

**何时不算**：表面相似实则因不同领域概念巧合一致——这种"虚假重复"如果合并会反而增加耦合。判断依据是"将来是否会一起变化"。

---

## 3. 过长函数（Long Function）

**症状**：

- 函数体超过约一屏（启发式而非硬性指标）
- 函数中有注释把代码"分段"
- 函数中含多层嵌套（>3 层）
- 一个函数同时做"取数、计算、格式化、输出"多件事

**嗅探问题**："做什么"和"怎么做"挤在一起；命名层级缺失。

**治理**（按场景）：
- 一段代码前有注释解释 → 提炼函数（注释内容→函数名）
- 函数中大量临时变量阻碍提炼 → 以查询取代临时变量
- 参数表已很长 → 引入参数对象 / 保持对象完整
- 庞大 switch / if-else 链 → 以多态取代条件表达式
- 嵌套循环每层做不同的事 → 拆分循环 后再提炼

**何时不算**：纯线性的初始化、配置注册、表驱动数据声明——这类"长但简单"的函数不需要拆。

---

## 4. 过长参数列表（Long Parameter List）

**症状**：参数 ≥4；或多个参数总成群结队出现（数据泥团 + 长参数列表）。

**嗅探问题**：参数列表是函数与世界的接口，过长意味职责不清或抽象缺失。

**治理**：
1. 多个参数总在一起 → 引入参数对象
2. 参数能从已传对象查得到 → 以查询取代参数 / 保持对象完整
3. 参数用作"分支开关" → 移除标记参数（拆成多个函数）
4. 多个函数共享同一组参数 → 函数组合成类（参数变字段）

**何时不算**：构造函数、纯函数式风格的"小依赖装配"接口。

---

## 5. 全局数据（Global Data）

**症状**：全局变量、单例、可变类静态变量；难以追踪是谁改了它。

**嗅探问题**：从代码任何角落可改，没有边界 → bug 可能从任何地方诞生。

**治理**：
1. 封装变量（先把全局数据藏到 getter/setter 后）
2. 把封装搬到模块/类内部，缩小作用域
3. 改成只读（程序启动后不再修改）

**何时不算**：真正的全局常量（编译期常量、配置常量）；运行期不可变的注入容器。

---

## 6. 可变数据（Mutable Data）

**症状**：同一变量在多处被改；更新逻辑分散；状态在对象生命周期里多次变换。

**嗅探问题**：A 处修改影响了 B 处的假设，bug 难以定位。

**治理**：
1. 封装变量（所有更新走少数函数）
2. 拆分变量（一个变量被复用做不同事 → 拆成多个）
3. 将查询函数和修改函数分离
4. 以查询取代派生变量（能算出来的就别存）
5. 将引用对象改为值对象（替换整个对象而非局部修改）

**何时不算**：作用域只有几行的局部变量；性能敏感的紧密循环里的累加器。

---

## 7. 发散式变化（Divergent Change）

**症状**：同一模块/类经常因**不同原因**在**不同方向**上被修改（"加数据库要改这 3 处，加金融工具要改那 4 处"）。

**嗅探问题**：模块承担了多个职责（违反单一职责）。

**治理**：
1. 两个方向有先后顺序 → 拆分阶段
2. 调用频繁 → 先建模块再 搬移函数
3. 函数内混合两类逻辑 → 先 提炼函数 再 搬移函数
4. 类形式 → 提炼类

**何时不算**：模块"看起来"有多个职责但它们其实总是一起变化（耦合内聚）。

---

## 8. 霰弹式修改（Shotgun Surgery）

**症状**：一个变化，需要散布到 N 个文件/类里改一点点；很容易漏改某处。

**嗅探问题**：本应聚拢的逻辑被过度切碎。

**治理**：
1. 搬移函数 / 搬移字段 把相关代码聚拢
2. 多函数操作相似数据 → 函数组合成类
3. 一系列函数转化数据结构 → 函数组合成变换
4. 已经分得太散 → 先 内联函数 / 内联类 拽回一处，再重新切分

**何时不算**：分散是出于跨层架构的合理需要（DTO、Adapter、Repository 各司其职）。

---

## 9. 依恋情结（Feature Envy）

**症状**：一个函数大量调用另一对象的 getter（"半打的取值函数"），而对自己所在对象的成员调用很少。

**嗅探问题**：函数住错了地方。

**治理**：
1. 整个函数依恋 → 搬移函数 到目标对象
2. 只有一段依恋 → 先 提炼函数 提那段，再 搬移函数
3. 函数用到多个对象数据 → 搬到"用到数据最多"的那个

**何时不算**：策略（Strategy）、访问者（Visitor）、Self Delegation 等模式刻意把行为与数据分离——这是为了对抗发散式变化。

---

## 10. 数据泥团（Data Clumps）

**症状**：三四个数据项总是绑在一起出现（多个类的字段、多个函数的参数）。

**嗅探问题**：缺失了一个本该存在的领域概念。

**治理**：
1. 字段成群 → 提炼类
2. 参数成群 → 引入参数对象 / 保持对象完整

**判定方法**：删掉其中一项，剩下的还有意义吗？没有 → 它们应该是一个对象。

---

## 11. 基本类型偏执（Primitive Obsession）

**症状**：用 `string`/`int`/`float` 表达"钱""日期""电话号""地理坐标""单位"等领域概念；`if (a < upper && a > lower)` 之类范围判断散落各处；string-typed 而非 strongly-typed。

**嗅探问题**：领域概念无家可归。

**治理**：
1. 以对象取代基本类型（`Money`、`PhoneNumber`、`Range`）
2. 控制条件行为的"类型码" → 以子类取代类型码 + 以多态取代条件表达式
3. 同时出现的多个基本类型 → 先识别为数据泥团，再 提炼类 / 引入参数对象

---

## 12. 重复的 switch（Repeated Switches）

**症状**：多处基于**同一个字段**做 switch / if-else 链；新增一种类型要找全所有 switch 改。

**嗅探问题**：类型分支没有变成多态。

**治理**：以多态取代条件表达式（每种类型一个子类，分支变为方法重写）。

**何时不算**：单一一处的 switch 不算坏味道（书第 2 版较第 1 版宽容许多）。坏的是"同样的 switch 出现在多处"。

---

## 13. 循环语句（Loops）

**症状**：命令式 `for` / `while`，循环体里混杂"过滤、变换、累加、副作用"多重职责。

**治理**：
1. 以管道取代循环（`filter` / `map` / `reduce`）
2. 一个循环做了几件事 → 先 拆分循环 再提炼

**何时不算**：性能敏感的内层循环（管道有抽象成本）；语言不支持函数式管道。

---

## 14. 冗赘的元素（Lazy Element）

**症状**：函数名几乎等于实现；类只裹着一个字段；"以为将来会变大"但一直没变大。

**治理**：
1. 内联函数 / 内联类
2. 处于继承体系中 → 折叠继承体系

**何时不算**：被设计为扩展点的接口/抽象类，即使当前只有一个实现。

---

## 15. 夸夸其谈通用性（Speculative Generality）

**症状**：为"也许将来会用到"准备的钩子、抽象类、参数、配置项；只在测试中被用到的代码。

**嗅探问题**：YAGNI 反例；增加心智负担又没付出回报。

**治理**：
1. 抽象类没作用 → 折叠继承体系
2. 不必要的委托 → 内联函数 / 内联类
3. 没人用的参数 → 改变函数声明 删掉
4. 仅测试用 → 移除测试 + 移除死代码

---

## 16. 临时字段（Temporary Field）

**症状**：类的字段只在某种特定情境下被赋值/使用；其他时候是 `null` / 未设置。

**嗅探问题**：对象的状态空间不一致；阅读者必须理解情境才能理解字段含义。

**治理**：
1. 提炼类（把临时字段和相关逻辑搬到新类）
2. 用到时才搬 → 搬移函数
3. "无值"的情况频繁 → 引入特例（Null Object）

---

## 17. 过长的消息链（Message Chains）

**症状**：`a.getB().getC().getD().getE()`；客户端依赖一长串导航结构。

**嗅探问题**：客户端与中间对象的内部结构耦合；中间任一对象变了，客户端跟着变。

**治理**：
1. 隐藏委托关系（让中间对象代为提供终点能力）
2. 看链终点的对象在做什么 → 提炼函数 那段使用代码，再 搬移函数 把它推到链中段

**何时不算**：流式 API（fluent builder）刻意设计的链式调用。

---

## 18. 中间人（Middle Man）

**症状**：类的接口里有一半函数都是"转发到另一个对象"的薄壳。

**嗅探问题**：封装过度——为了藏一层，引入了无价值的代理。

**治理**：
1. 移除中间人（让客户端直接面对真正的对象）
2. 委托不多 → 内联函数 把转发塞回调用方
3. 中间人还有别的行为 → 以委托取代超类 / 以委托取代子类

---

## 19. 内幕交易（Insider Trading）

**症状**：模块间频繁地传递内部细节、互相窥探。

**治理**：
1. 搬移函数 / 搬移字段 减少私下交流
2. 共同关心的东西 → 提一个新模块管它
3. 子类对超类知道得过多 → 以委托取代子类 / 以委托取代超类

---

## 20. 过大的类（Large Class）

**症状**：字段多；方法多；前缀/后缀相同的字段成群（`startDate`、`endDate`、`startTime`、`endTime`...）；类内多个变量重复 → 重复代码连带产生。

**治理**：
1. 同前缀/同关注点字段 → 提炼类
2. 适合做子类 → 提炼超类 或 以子类取代类型码
3. 不同使用者只用其子集 → 按使用者拆类
4. 有 5 个 100 行函数 → 拆成 5 个 10 行 + 10 个 2 行的"双层结构"

---

## 21. 异曲同工的类（Alternative Classes with Different Interfaces）

**症状**：两个类做相似的事，但接口不一致——你想用多态替换，但签名对不上。

**治理**：
1. 改变函数声明 对齐签名
2. 反复 搬移函数 把一致的行为推进去
3. 重复出现 → 提炼超类

---

## 22. 纯数据类（Data Class）

**症状**：类只有字段和 getter/setter，没有行为；行为散落在各使用者那里（伴随依恋情结）。

**治理**：
1. public 字段 → 封装记录
2. 不应被改的字段 → 移除设值函数
3. 找到它的 getter/setter 被调用的地方 → 把那些行为 搬移函数 进来；不能整体搬就先 提炼函数

**何时不算**：作为函数返回值的中转结构（如 拆分阶段 的产物）；不可变值对象。

---

## 23. 被拒绝的遗赠（Refused Bequest）

**症状**：子类不需要超类的某些字段/方法；甚至不愿意支持超类的接口契约。

**治理**：
1. 不用某些字段/方法 → 字段下移 / 函数下移
2. 拒绝接口契约 → 以委托取代超类 / 以委托取代子类（彻底脱离继承）

**何时不算**：轻度的"不全用"是可接受的——不是每个父类都得纯抽象。强度判断点是"是否拒绝接口契约"。

---

## 24. 注释（Comments）

**注释不是坏味道**——但它常被当除臭剂用。

**症状**：长长的注释解释一段晦涩的代码"在做什么"。

**治理**：
1. 注释解释一块代码 → 提炼函数（注释内容→函数名）
2. 函数已存在但需要注释 → 改变函数声明 改名
3. 注释是"系统需求规格" → 引入断言（用代码表达约束）

**注释什么时候是好的**：

- 记录"为什么"这么做（决策理由、权衡）
- 标记"我不确定的地方"
- 引用外部文档（标准、issue 编号）
- API 文档（面向使用方的契约说明）

> 当你感觉需要撰写注释时，请先尝试重构，试着让所有注释都变得多余。— Fowler
