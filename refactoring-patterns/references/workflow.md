# 重构工作流：守门、节奏、回滚

本文回答的是"怎么做才安全、可控、可持续"。坏味道与手法是"做什么"，工作流是"怎么做"。

## 1. 测试守门：没测试就别动手

> 一套测试就是一个强大的 bug 侦测器，能够大大缩减查找 bug 所需的时间。 — Fowler

### 自测试代码三条标准

1. **完全自动化** — 测试自检结果，绿/红一目了然，不要靠人盯输出
2. **跑得够频繁** — 编辑→保存→测试 应在秒级；CI 是兜底，不是主战场
3. **覆盖到要重构的部分** — "这段代码有没有被测试覆盖"是动手前的硬性检查

### 没有测试的遗留代码：先补特征测试

**特征测试（characterization test）** 不是"我希望它怎样"，而是"它现在到底是怎样"：

1. 给目标函数/类喂典型输入，记录当前输出（哪怕输出是错的）
2. 把"当前输出"写进 assert，作为基线
3. 接下来重构时，这些 assert 会守住"行为不变"
4. 重构完成后再单独评估那些"输出错的地方"是否要修

> 关键：特征测试不评判对错，只锁定行为。重构期间它就是合同。

### 应该写多少测试？

> 少许测试往往就足以带来惊人的收益。 — Fowler

不必追求 100% 覆盖。优先覆盖：

- 你**即将**重构的代码路径
- 边界条件（空集、零、负数、Unicode、跨时区日期）
- 历史上 bug 反复出现的地方
- 接口边界（公共 API、模块边界）

> 内部细节不必测。重构会改它们；测它们会让你不敢动。

## 2. 两顶帽子（Two Hats）

任何时刻只戴一顶：

| 帽子 | 允许 | 禁止 |
|---|---|---|
| **加新功能 / 修 bug** | 改行为；写新测试；改测试 | 顺手清理代码 |
| **重构** | 改结构；运行测试 | 改外部行为；写新测试（除非补遗漏的特征测试）；改测试（除非接口变了） |

### 切换前，先提交

```
[Feature]    -> commit "feat: ..."
[Refactor]   -> commit "refactor: extract ..."
[Refactor]   -> commit "refactor: rename ..."
[Feature]    -> commit "feat: ..."
```

混在一起的提交在 review 时是噩梦：审阅者无法分辨"这一行是为了改行为，还是为了改结构"。

## 3. 小步前进的节奏

> 小步前进能让我走得更快，因为这些小步骤能完美地彼此组合，而且——更关键的是——整个过程中我不会花任何时间来调试。 — Fowler

### 经验值

- 一步做完，应该 ≤5 分钟
- 一步做完到下次跑测试，应该 ≤30 秒
- 测试一红，先 revert（哪怕你"差不多看出问题在哪了"）；revert 后把步子拆得更小再来

### 何时步子可以大一点

- 你对该手法非常熟、对该代码非常熟
- 测试覆盖密、跑得快
- 出错代价低（实验分支、个人项目）

### 何时必须步子很小

- 不熟悉代码
- 测试稀疏
- 多人协作分支
- 关键路径

## 4. 回滚策略

```
git stash                # 临时存放，用于切到 main 跑全量测试再回来
git revert <sha>         # 把某次提交反向地再提交一次（最常用，最安全）
git reset --hard HEAD    # 仅当未推送、且无你想保留的工作区改动
```

> 永远不要 `git reset --hard` 已推送的提交。`revert` 是默认武器。

每次提交都要写得"独立可 revert"——这是检验小步前进是否合格的最直接标准。

## 5. 重构 vs 性能优化

| 维度 | 重构 | 性能优化 |
|---|---|---|
| 行为 | 不变 | 不变 |
| 可读性 | 必须提升 | 可能下降 |
| 测试要求 | 全绿 | 全绿 + 性能基线 |
| 时机 | 任何时候 | profiler 指出热点之后 |

> 大多数情况下，重构会让代码稍微慢一点（多一层间接）。如果性能不是热点路径，**忽略它**。如果是热点，先用 profiler 度量，再针对性优化。"重构会拖慢程序"是被反复证伪的迷思。

## 6. 长期 / 大型重构：Branch By Abstraction

替换一个被广泛使用的库、把巨型模块拆成多个、收紧一层架构边界——这种重构不可能"一周做完"。

**不要**：拉一个长期分支，闭关一个月，最后大爆炸式合并。

**做法（Branch By Abstraction）**：

1. 在调用方与"被替换者"之间插入一层薄抽象（接口/适配器）
2. 让这层抽象兼容**新旧两种实现**
3. 一个调用点一个调用点地切到新实现
4. 全部切完之后，删掉旧实现和兼容层

每一步都让 main 分支保持可工作。Feature flag / 开关切换可以加在抽象层。整个过程可能持续数周，但每天都能合并、每天都能发布。

## 7. 提交信息规范

```
refactor: extract calculateTotal()
refactor: rename `data` -> `customer`
refactor: replace temp with query in printOwing
refactor: introduce parameter object Range
refactor: replace conditional with polymorphism (Bird type)
```

要点：

- 用动词起头，描述这一次提交做了什么具体手法
- 手法名直接出现（提炼函数 / extract function 等），便于 review 与日后 grep 历史
- 只描述**结构层面**的改动；不写"修复 bug"或"添加功能"——那是另一顶帽子的事
- 一次提交应该对应一个手法的一次完整应用（或同一手法的几次小应用）

## 8. 何时不应该重构

- **要被重写**的代码：重构是为修改而铺路，将被丢弃的代码不需要
- **稳定运行、不需要修改、且被良好封装**的代码：哪怕内部丑，留它
- **关键路径 + 没有测试 + 短期补不上特征测试**：先补测试或评估改写
- **临近发布**：戴好"加功能"那顶帽子；重构记入下一个迭代
- **不熟悉的代码 + 没有同事可以问 + 没有测试**：理解先于改动

## 9. 怎么对经理说

> 重构的唯一目的就是让我们开发更快，用更少的工作量创造更大的价值。 — Fowler

当被问到"为什么花时间重构"时：

- **错误回答**："为了代码更整洁 / 更优雅"——这把重构变成了道德/审美问题
- **正确回答**："为了让接下来这个新功能能在 X 天内完成而不是 Y 天"——把重构定位为加速器

如果经理不懂技术、又用进度施压：

> 我们是专业人士。如果重构是把任务做完最快的方式，那就重构。你不需要每写一个 if 就先报备。

## 10. 重构 ≠ 结构调整

最后澄清术语：

- **重构（refactoring）** = 不改变可观察行为 + 小步前进 + 测试守门
- **结构调整（restructuring）** = 任意形式的代码重新组织，可能改变行为，可能不可工作几天

把任意形式的清理叫"重构"是行业里常见的滥用。本技能严格区分这两者。如果你打算做的事不满足重构的两条硬约束，那就别叫它重构——叫"重写"或"返工"，并按对应的风险等级管理它。
