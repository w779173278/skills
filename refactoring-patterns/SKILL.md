# 以坏味道驱动的代码重构

本技能把 Fowler《重构》第 2 版浓缩为一张"症状→手法→机制→守门"的工作图。先识别坏味道，再选手法，再以小步可逆的方式落地，每一步都有测试守门。

## 何时使用本技能

- 要在添加新功能或修 bug **之前**做"预备性重构"，让接下来的修改变容易
- 在阅读代码时被某段卡住、需要"帮助理解的重构"
- PR review 时发现结构问题、想给出可落地的重构建议
- 顺手清理一段经过的脏代码（"捡垃圾式重构"）
- 想做一次有计划的中型重构（库替换、模块拆分、抽象引入）

**何时不使用**：

- 即将整体重写的代码：重写比重构便宜，跳过
- 不需要被理解或修改的丑代码（已被良好封装、稳定运行）：留着
- 没有任何测试、且短期内补不上特征测试的关键路径：先补测试再说
- 进度紧到只剩一天的发布前：戴好"加新功能"那顶帽子，重构留给下一轮

## 核心原则（不可妥协）

1. **不改变可观察行为** — 重构不是改需求。bug 在重构前后应同样存在（除非你顺手把它修了）
2. **两顶帽子，戴清楚一顶** — 同一时刻只做一件事：要么"加功能/修 bug"（可写测试、可改行为），要么"重构"（不写新测试、不改行为）。中途切换前先提交
3. **小步前进、随时可停** — 每一步都让代码处于可工作状态。"重构停下来一两天代码不可用"，那不是重构
4. **测试守门** — 没有自测试代码就别动手；测试不全则先补特征测试（characterization test）锁定既有行为
5. **命名表达意图** — 名字描述"做什么"而非"怎么做"。想不出好名字，往往是设计问题
6. **每一步都可一键回滚** — 任何改动应该 `git revert` 一次就能撤销

## 三种重构时机

| 时机 | 触发 | 帽子 | 提交策略 |
|---|---|---|---|
| 预备性重构 | 准备加功能/修 bug 时发现现状结构碍事 | 重构帽 | 先重构提交，再切换帽子加功能 |
| 帮助理解的重构 | 读代码读不懂、需要把理解"植入代码" | 重构帽 | 改名、提炼小函数即提交 |
| 捡垃圾式重构 | 顺路看到的小污点 | 重构帽（短戴） | 当场清理 ≤5 分钟内能搞定的；超出就记在便笺里 |

> **Kent Beck**："每次要修改时，首先令修改很容易（警告：这件事有时会很难），然后再进行这次容易的修改。"

## 工作流（每次重构循环）

```
1. 识别坏味道           ← 见下表，或参 references/smells.md
2. 选定一个具体手法      ← 见 references/catalog.md
3. 确认有测试守门         ← 没有则先补；见 references/workflow.md
4. 执行最小一步          ← 每步只动一件事
5. 跑测试 → 全绿         ← 红了立即 revert，把步子拆得更小
6. 提交                  ← 信息含手法名，例 "refactor: extract calculateTotal()"
7. 继续下一步 / 切换帽子 / 收手
```

> 提示：重构提交与功能提交分不分开看团队习惯。Fowler 倾向不强制分离（成本大于收益）；但**同一个提交里不要混功能改动与重构改动**——这是底线。

## 24 种坏味道 → 推荐手法（决策表）

> 详细的"识别信号 / 排查思路 / 反例"见 [references/smells.md](references/smells.md)。

| # | 坏味道 | 主要识别信号 | 推荐手法（按优先级） |
|---|---|---|---|
| 1 | 神秘命名（Mysterious Name） | 看名字猜不出意图；要读实现才能用 | 改变函数声明、变量改名、字段改名 |
| 2 | 重复代码（Duplicated Code） | 多处出现相同/相似表达式 | 提炼函数；移动语句对齐后再提炼；同一继承体系用函数上移 |
| 3 | 过长函数（Long Function） | 函数体超过约一屏；含注释分块；含多层嵌套 | 提炼函数（最常用）；以查询取代临时变量；引入参数对象；以多态取代条件表达式 |
| 4 | 过长参数列表（Long Parameter List） | 参数 ≥4，或参数总成群结队 | 引入参数对象；保持对象完整；以查询取代参数；移除标记参数 |
| 5 | 全局数据（Global Data） | 全局变量、单例、可变类变量从各处被读写 | 封装变量；进一步搬移到模块/类内；变成只读 |
| 6 | 可变数据（Mutable Data） | 同一变量在多处被改；复杂的更新逻辑 | 封装变量；拆分变量；查询/修改分离；以查询取代派生变量；将引用对象改为值对象 |
| 7 | 发散式变化（Divergent Change） | 同一模块因不同原因在不同方向被改 | 拆分阶段；搬移函数；提炼类 |
| 8 | 霰弹式修改（Shotgun Surgery） | 一个变化散落到 N 个文件/类里 | 搬移函数、搬移字段把相关代码聚拢；函数组合成类；必要时先内联再重新提炼 |
| 9 | 依恋情结（Feature Envy） | 一个函数大量调用另一对象的取值 | 搬移函数；若只有一段是这样，先提炼函数再搬移 |
| 10 | 数据泥团（Data Clumps） | 多个数据项总是结伴出现 | 提炼类；引入参数对象；保持对象完整 |
| 11 | 基本类型偏执（Primitive Obsession） | 用 string/int 表达"钱""日期""电话号"等领域概念 | 以对象取代基本类型；以子类取代类型码 + 以多态取代条件表达式 |
| 12 | 重复的 switch（Repeated Switches） | 多处基于同一字段做 switch/if-else 链 | 以多态取代条件表达式 |
| 13 | 循环语句（Loops） | 命令式循环里混杂多重职责 | 以管道取代循环；先拆分循环再提炼 |
| 14 | 冗赘的元素（Lazy Element） | 类/函数没承担应有的额外结构价值 | 内联函数；内联类；折叠继承体系 |
| 15 | 夸夸其谈通用性（Speculative Generality） | 为"也许将来"准备的钩子/抽象/参数 | 折叠继承体系；内联函数/类；改变函数声明去掉无用参数；移除死代码 |
| 16 | 临时字段（Temporary Field） | 字段只在某种情境下被赋值/使用 | 提炼类；搬移函数；引入特例（Null Object） |
| 17 | 过长的消息链（Message Chains） | `a.b().c().d().e()` 之类的链 | 隐藏委托关系；先提炼函数再搬移到链中段 |
| 18 | 中间人（Middle Man） | 类的接口大量委托给另一个对象 | 移除中间人；少量委托用内联函数；必要时以委托取代超类/子类 |
| 19 | 内幕交易（Insider Trading） | 模块间过度共享内部细节 | 搬移函数/字段；隐藏委托关系；以委托取代继承 |
| 20 | 过大的类（Large Class） | 字段多、方法多、关注点不单一 | 提炼类；提炼超类；以子类取代类型码 |
| 21 | 异曲同工的类（Alternative Classes ≠ Interfaces） | 两类做相似事但接口不一致 | 改变函数声明对齐签名；搬移函数对齐行为；提炼超类 |
| 22 | 纯数据类（Data Class） | 只有 getter/setter，没有行为 | 封装记录；移除设值函数；用搬移函数把行为搬进来 |
| 23 | 被拒绝的遗赠（Refused Bequest） | 子类不需要超类的字段/方法 | 字段下移、函数下移；严重时以委托取代超类/子类 |
| 24 | 注释（Comments）| 注释在替代糟糕代码道歉 | 提炼函数（注释 → 函数名）；改变函数声明；引入断言 |

## 25 个核心重构手法（速查）

> 完整的"做法/范例/常见陷阱"见 [references/catalog.md](references/catalog.md)。这里只列触发条件，便于在阅读上表后快速定位。

**第一组（基础）**：提炼函数、内联函数、提炼变量、内联变量、改变函数声明、变量改名、封装变量、引入参数对象、函数组合成类、函数组合成变换、拆分阶段。

**封装**：封装记录、封装集合、以对象取代基本类型、以查询取代临时变量、提炼类、内联类、隐藏委托关系、移除中间人、以函数取代字段。

**搬移**：搬移函数、搬移字段、以查询取代参数、移除标记参数、保持对象完整、将查询函数和修改函数分离、以工厂函数取代构造函数。

**条件逻辑**：分解条件表达式、合并条件表达式、以卫语句取代嵌套条件、以多态取代条件表达式、引入特例、引入断言。

**API**：将查询函数和修改函数分离、函数参数化、移除标记参数、保持对象完整、以查询取代参数、移除设值函数、以工厂函数取代构造函数、以命令取代函数 / 以函数取代命令。

**继承**：函数上移、字段上移、构造函数本体上移、函数下移、字段下移、以子类取代类型码、移除子类、提炼超类、折叠继承体系、以委托取代子类、以委托取代超类。

## 快速决策：要不要现在重构？

```
新功能/修 bug 即将开始？
├─ 是 → 现状碍不碍事？
│         ├─ 碍事 → 预备性重构（先重构再加功能）
│         └─ 不碍 → 直接做，戴新功能帽
└─ 否 → 这段代码我会再读/再改吗？
          ├─ 是 → 帮助理解的重构（小步：改名、提炼）
          └─ 否 → 不重构（即使丑也不动；隐藏在 API 后即可）
```
